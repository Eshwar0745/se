# Quick overview of the important Maven concepts you‚Äôll see

* `groupId` ‚Äî project group (like company/org).
* `artifactId` ‚Äî module name.
* `version` ‚Äî project version (e.g., `0.0.1-SNAPSHOT`).
* `packaging` ‚Äî `pom`, `jar`, `war` etc. Parent must be `pom` to contain modules.
* `<modules>` in parent `pom` ‚Äî lists child modules.
* `<parent>` in child `pom` ‚Äî makes child inherit groupId/version, plugin config, and dependencyManagement.
* `dependency` ‚Äî add one child as a dependency of another using groupId/artifactId/version.
* `mvn -pl <module> -am` ‚Äî build a module and also build modules it depends on.

# Step-by-step (Eclipse + brief CLI tips)

### 1) Create parent project in Eclipse

1. Open Eclipse ‚Üí **File ‚Üí New ‚Üí Other ‚Üí Maven ‚Üí Maven Project** ‚Üí Next.
2. Check: **Create a simple project (skip archetype selection)** ‚Üí Next.
3. Fill values:

   * `Group Id`: `KMIT`
   * `Artifact Id`: `MultiModule`
   * `Packaging`: **select `pom`** (important!)
   * `Name`: `Multimodule creation`
   * `Description`: `Sample multimodule with parent child hierarchy`
4. Click **Finish**.
   **Why:** packaging `pom` means this project acts as a parent/aggregator and does not produce a JAR/WAR itself.

---

### 2) Create child modules (module1 = JAR)

1. Right-click the `MultiModule` folder ‚Üí **New ‚Üí Maven Module** (or New ‚Üí Other ‚Üí Maven ‚Üí Maven Module).
2. Choose **Create a simple project (skip archetype selection)** ‚Üí Next.
3. `ArtifactId`: `MultiModuleChild1` ‚Üí Next ‚Üí Finish.
4. Eclipse creates `MultiModuleChild1` folder and its own `pom.xml`.
   **Why:** a simple child defaults to `jar` packaging ‚Äî good for libraries/core.

---

### 3) Create child module2 (a webapp)

1. Right-click `MultiModule` ‚Üí **New ‚Üí Maven Module**.
2. Search archetypes: choose `maven-archetype-webapp` (versions 1.1 or 1.5) ‚Üí Next ‚Üí Finish ‚Üí confirm `Y` in console.
3. Now `MultiModuleChild2` (webapp) appears with a `pom.xml` and webapp structure (`src/main/webapp`).
   **Why:** web archetype gives standard webapp layout and `war` packaging.

---

### 4) Make parent aware of its modules (add `<modules>` to parent `pom.xml`)

Eclipse sometimes updates parent automatically, but ensure parent `pom.xml` contains:

```xml
<project ...>
  <modelVersion>4.0.0</modelVersion>
  <groupId>KMIT</groupId>
  <artifactId>MultiModule</artifactId>
  <version>0.0.1-SNAPSHOT</version>
  <packaging>pom</packaging>

  <modules>
    <module>MultiModuleChild1</module>
    <module>MultiModuleChild2</module>
  </modules>
</project>
```

**Why:** `<modules>` is required for Maven to treat children as part of the aggregated build.

---

### 5) Add parent reference in child poms (so children inherit)

In each child `pom.xml`, add a `<parent>` block (so you don‚Äôt repeat `groupId/version` and inherit config):

```xml
<project ...>
  <parent>
    <groupId>KMIT</groupId>
    <artifactId>MultiModule</artifactId>
    <version>0.0.1-SNAPSHOT</version>
    <!-- <relativePath>../pom.xml</relativePath> optional -->
  </parent>

  <artifactId>MultimoduleChild1</artifactId>
  <packaging>jar</packaging>
  ...
</project>
```

**Why:** inheritance gives consistent group/version and centralized plugin/dependency config.

---

### 6) Link two modules (child2 depends on child1)

In `MultiModuleChild2`'s `pom.xml` add dependency:

```xml
<dependencies>
  <dependency>
    <groupId>KMIT</groupId>
    <artifactId>MultiModuleChild1</artifactId>
    <version>0.0.1-SNAPSHOT</version>
  </dependency>
  <!-- other deps -->
</dependencies>
```

**Why:** This tells child2 to use classes/artifacts from child1. Because both are in the same multi-module project, Maven can build them together.

**Tip:** If you use `<dependencyManagement>` in the parent, children can omit `<version>` in dependency declarations and inherit versions centrally.

---

### 7) Build and test (CLI & Eclipse)

* From parent project root (contains parent `pom.xml`):

  * `mvn clean install` ‚Üí builds parent and all modules and installs them to your local repo.
* To build only module2 and its dependencies:

  * `mvn -pl MultiModuleChild2 -am clean package`
    (`-pl` = projects list, `-am` = also make dependencies)
* In Eclipse: Right-click parent ‚Üí **Run As ‚Üí Maven install** (same effect).

**Why:** `install` places artifacts in your local Maven repo for other projects to consume; `package` creates the JAR/WAR.

# Example better parent with dependencyManagement and pluginManagement

```xml
<project ...>
  <packaging>pom</packaging>
  <groupId>KMIT</groupId>
  <artifactId>MultiModule</artifactId>
  <version>0.0.1-SNAPSHOT</version>

  <modules>
    <module>MultiModuleChild1</module>
    <module>MultiModuleChild2</module>
  </modules>

  <dependencyManagement>
    <dependencies>
      <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-core</artifactId>
        <version>5.3.30</version>
      </dependency>
      <!-- centralize common versions -->
    </dependencies>
  </dependencyManagement>

  <build>
    <pluginManagement>
      <!-- common plugin versions/config for children -->
    </pluginManagement>
  </build>
</project>
```

**Why:** keeps versions consistent and reduces duplication.

# Common pitfalls & fixes

* **Parent packaging not `pom`** ‚Üí modules can‚Äôt be added. Always set `packaging` to `pom`.
* **Child not listed under `<modules>`** ‚Üí parent build won‚Äôt pick it up. Add it manually or recreate module under parent.
* **Version mismatch** ‚Üí ensure child dependency uses same `version` as parent or use `dependencyManagement`.
* **Trying to add module inside JAR child** ‚Üí a child with `jar` packaging shouldn‚Äôt act as aggregator; create modules under the parent instead.
* **Eclipse m2e out of sync** ‚Üí right-click project ‚Üí **Maven ‚Üí Update Project**.

# Handy commands you‚Äôll use repeatedly

* `mvn clean install` ‚Äî build everything and install to local repo.
* `mvn -pl <module> -am package` ‚Äî build a module and modules it depends on.
* `mvn -T 1C clean install` ‚Äî parallel build (speeds up big multimodule builds).
* `mvn dependency:tree` ‚Äî see dependency graph (useful to debug transitive deps).

---

# Short checklist (so you don‚Äôt miss anything)

1. Parent `pom` created with `packaging=pom`.
2. Child modules created under parent folder.
3. Parent `<modules>` lists child folders.
4. Child `pom`s include `<parent>` block and correct `artifactId`/`packaging`.
5. Child2 includes `<dependency>` on child1 if it needs it.
6. `mvn clean install` from parent builds all.

# **9. Jenkins Pipeline Using Script ‚Äî Ultra-Short Steps**

### **1. Create a Pipeline Job**

1. Jenkins ‚Üí **New Item**
2. Enter name: `pipeline_script`
3. Select **Pipeline** ‚Üí OK
### **2. Add Pipeline Script**

Pipeline ‚Üí Definition: **Pipeline script**
Paste a simple script:

```groovy
pipeline {
    agent any
    tools{
        maven 'MAVEN_HOME'
    }
    stages {
        stage('git repo & clean') {
            steps {
                bat "rmdir  /s /q xes || exit 0"
                bat "git clone https://github.com/Eshwar0745/xes.git"
                bat "mvn clean -f xes"
            }
        }
        stage('install') {
            steps {
                bat "mvn install -f xes"
            }
        }
        stage('test') {
            steps {
                bat "mvn test -f xes"
            }
        }
        stage('package') {
            steps {
                bat "mvn package -f xes"
            }
        }
    }
}
```

---

### **3. Save Job**

Click **Apply ‚Üí Save**.

5. Key pair: **Create ‚Üí RSA ‚Üí .pem ‚Üí Download**

   * Add **Custom TCP (8081)** ‚Üí 0.0.0.0/0

8. Launch

maven
